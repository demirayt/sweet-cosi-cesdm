Generation

GenerationUnitEnergyAssetInstance
A deployed generation asset with stable identity and carrier/resource relations. Technology-specific construction and configuration data are represented by static technical views; operational values belong in Generation.DispatchView and Generator.PowerFlowView. Technology classification and shared defaults are expressed through hasTechnology rather than through asset subclasses.

ExternalSupplyEnergyAssetInstance
An external supply point that injects a single output carrier into an EnergyBus and acts as a slack source for the modelled system.
An ExternalSupply represents energy flowing into the model boundary from outside the explicitly modelled perimeter — e.g. an import from a neighbouring market zone, a grid connection of undefined origin, or a reference slack bus used as the angular and power balance reference in load-flow and dispatch models.
As a slack source it absorbs all power imbalances within a connected island (i.e. its injection is solved, not fixed). The supply price attribute defines the marginal cost of this balancing energy.
Differences from GenerationUnit:
- Capacity is optional (default: uncapacitated).
- The supply is not a physical asset within the modelled system; it
represents a system boundary condition.
- In AC power flow the connected bus should be assigned
powerflow_bus_type = slack on its ElectricalBus.PowerFlowView.
- In optimisation the ExternalSupply.DispatchView.supply_price defines
the cost of imbalance energy.

One ExternalSupply with is_slack = true should exist per connected electrical island.
Topological connection is expressed in SinglePort.TopologyView.atNode. Economic parameters are expressed in ExternalSupply.DispatchView.

Conversion

ConversionUnitEnergyAssetInstance
A specific deployed conversion unit that transforms one or more input carriers into one or more output carriers, potentially spanning multiple CarrierDomains.
Carrier topology is expressed exclusively through ConversionPort entities. Each ConversionPort records its bus connection (atNode), carrier (hasCarrier), flow direction, and flow coefficient, and points back to this unit via belongsToUnit. Ports are therefore discoverable without any forward relation on the unit itself.
This design makes ConversionPort the single source of truth for carrier information. Do NOT set hasInputCarrier or hasOutputCarrier directly on a ConversionUnit — doing so creates a second, potentially inconsistent source that import tools would have to reconcile. If a quick carrier annotation is needed for tooling that does not support the full port model (e.g. PyPSA import without gas network), set the carriers on the GenerationUnit subclass instead, or create a minimal single-port ConversionPort.
Operational and economic parameters belong in Conversion.DispatchView.
Examples: CHP plant, electrolyser, heat pump, fuel cell, methanation reactor, P2X converter.

NetworkNodeabstractSemanticEntity
Abstract base class representing a topological connection point within a
CarrierDomain where the domain's EnergyCarrier can be injected, withdrawn,
exchanged, or balanced between connected assets.

NetworkNode is explicitly NOT a SystemAsset. A node is a mathematical and
topological primitive — a point of enforced carrier balance — not a deployable
or operable unit. In network analysis (power flow, dispatch optimisation, gas
network simulation) nodes are the "where", while assets are the "what".

If a physical object (a substation building, a compressor station) needs to be
modelled as an asset in addition to being a network node, create a separate
EnergyAssetInstance referencing the same node via a relation; do not conflate
the two concepts on one entity.

Spatial coordinates (latitude, longitude, elevation) are carried on BusLocationView,
not on the NetworkNode entity itself. BusLocationView is the single source of truth
for spatial data. This avoids the ambiguity of two coordinate sources that import
tools would have to reconcile. For workflows that only need a quick coordinate
annotation without GIS export, create a minimal BusLocationView with only lat/lon
set — it is a lightweight entity.

Carrier-specific operating-point attributes (voltage, pressure, temperature, head)
belong on typed subclasses:
ElectricalBus → nominal_voltage [kV]
GasBus → nominal_pressure [bar]
HydrogenBus → nominal_pressure [bar]
HeatBus → nominal_temperature [°C], nominal_pressure [bar]
WaterBus → nominal_head [m]

Topological connections (which assets attach to this node) are expressed through
NetworkTopologyView subclasses (SinglePort.TopologyView, TwoPort.TopologyView), never
directly on the node entity.

ConversionPortPort
A single input or output port of a ConversionUnit in the Tier 2 MIMO (multiple-input multiple-output) representation.
Each ConversionPort is a first-class entity that records:
- which Bus it connects to (atNode relation)
- which EnergyCarrier flows at this port (hasCarrier relation)
- the flow direction (input / output / bidirectional)
- the flow coefficient relative to the unit's reference port
- optionally a time-varying flow coefficient profile

Carrier-specific efficiencies or output ratios are represented at port level through flow_coefficient. For example, for a CHP with a gas reference input, the electrical and heat efficiencies are the flow coefficients of the electricity and heat output ports, not attributes of the unit-level Conversion.DispatchView.
The reference port (is_reference_port = true) defines the scale: flow_coefficient = 1.0 for the reference port by definition. All other ports express their flow as a signed ratio to the reference port flow. Positive = injection into the Bus (output); negative = withdrawal from the Bus (input).
Exactly one ConversionPort per ConversionUnit must be the reference port. It is typically the primary fuel or energy input.
ConversionPort is a domain entity, not a view. It describes the physical port structure of the unit, which is a topology fact that exists independently of any analysis.

ConversionPort is a structural port/interface, not an EnergyAssetInstance. It belongs to a ConversionUnit or component and can connect to NetworkNode instances.

ProfileSemanticEntity
Represents a named time series of scalar values associated with an energy system entity (generator resource potential, demand shape, inflow volume, price signal, etc.).
The Profile entity carries only interpretation metadata as attributes. The numeric payload — a float64 array of length equal to the TimestampSeries length — is stored exclusively in the HDF5 file under /profiles/<entity_id>/values. This strict separation keeps the YAML model human-readable and version-controllable while delegating bulk numeric storage to HDF5.
Three profile types are supported:
as_SI Values are in absolute SI units (e.g. MW,
MWh, m3/s). Multiply by 1.0 to use directly.
as_capacity_factor Dimensionless fractions [0, 1]. Multiply by
the asset nominal_power_capacity to obtain
absolute power values.
as_normalized_annual_energy Dimensionless fractions that sum to 1.0
over the full year. Multiply by the asset
annual_energy_demand or annual_resource_potential
to obtain absolute energy values per timestep.
